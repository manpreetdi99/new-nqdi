Advanced Rebase Workflows

Understanding Rebase vs Merge

Rebase replays commits from one branch onto another, creating a linear history. Unlike merge, which creates a merge commit, rebase rewrites your branch's history.

Golden Rule: Never rebase commits that have been pushed to a shared repository, unless you coordinate with your team. Rebase rewrites history, which will cause conflicts for anyone who has pulled those commits.

Basic Rebase: Linear History

$ git rebase main

# Replays current branch's commits on top of main

Interactive Rebase (rebase -i)

The most powerful Git command for rewriting history. Allows you to reorder, squash, edit, or delete commits.

$ git rebase -i HEAD~5

pick a1b2c3d Fix typo
squash b2c3d4e Add validation
fixup c3d4e5f Refactor parser
reword d4e5f6g WIP: debugging
drop e5f6g7h Old commit

Interactive Rebase Commands

Command Effect Use Case
pick Keep commit as-is Default
reword Keep commit, edit message Fix typos
squash Merge into previous Combine commits
fixup Squash, discard message WIP commits
drop Delete entirely Remove bad commits
exec Run shell command Test each commit

Rebase with --onto

Rebase a branch onto a different base. Useful for moving branches or extracting commits.

$ git rebase --onto main feature~2 feature

# Takes last 2 commits of feature and replays on main

Deleting Commits: The Definitive Guide

Delete the Last Commit

Option 1: Soft Reset (Keep Changes)

$ git reset --soft HEAD~1

# Removes commit, keeps staged changes

Option 2: Mixed Reset (Default)

$ git reset HEAD~1

# Removes commit, unstages changes

Option 3: Hard Reset (Discard Everything)

$ git reset --hard HEAD~1

# ⚠️ DANGEROUS: Discards all changes!
Recovery Possible: Unpushed commits live in reflog for ~30 days. See the Reflog section.

Delete Multiple Recent Commits

$ git reset --soft HEAD~3

# Removes last 3 commits, keeps changes

Delete a Commit in the Middle

$ git rebase -i HEAD~3

# Mark commit as "drop" to delete it

Delete Using Filter-branch

$ git filter-branch --tree-filter 'rm -f secret-key.txt' HEAD

# Removes secret-key.txt from entire history

Stash: Advanced Techniques

Why Stash Matters

Stash temporarily saves uncommitted changes without committing them. Essential for switching branches without losing work.

Basic Stash Operations

$ git stash
# Saves all changes

$ git stash list
stash@{0}: WIP on feature/auth: a1b2c3d

$ git stash pop
# Restores and removes from list

$ git stash apply
# Restores but keeps in list

Stash with Meaningful Names

$ git stash push -m "WIP: API refactoring"

$ git stash list
stash@{0}: WIP: API refactoring

Partial Stash: Save Only Some Files

$ git stash push src/api.js src/utils.js -m "API changes"

# Only stashes specified files

Stash Untracked Files

$ git stash -u

# -u = --include-untracked

Stash Only Staged Changes

$ git stash --staged

# Stashes only git add'ed changes

Interactive Stash Selection

$ git stash push -p

# -p = --patch: asks for each hunk

View Stash Content

$ git stash show -p stash@{0}

# Shows full diff of the stash

Interactive Rebase: Deep Dive

Setup and Workflow

$ git rebase -i HEAD~5

# Opens editor with last 5 commits

Example: Clean Messy History

Scenario: Clean messy feature branch
Before:
a1b2c3d Add form validation
b2c3d4e Fix typo
c3d4e5f WIP: debugging
d4e5f6g More debugging
e5f6g7h Refactor form
f6g7h8i More fixes

After rebase -i:
pick a1b2c3d Add form validation
fixup b2c3d4e Fix typo
fixup c3d4e5f WIP: debugging
fixup d4e5f6g More debugging
pick e5f6g7h Refactor form
fixup f6g7h8i More fixes

Result: 2 clean commits instead of 6

Editing Commit Content During Rebase

$ git rebase -i HEAD~3

# Mark as "edit"
edit a1b2c3d Partial feature

# Rebase stops at a1b2c3d
$ git reset --soft HEAD~1
$ git add src/api.js
$ git commit -m "API changes only"
$ git rebase --continue

Reorder Commits

Change commit order by moving lines in the editor.

$ git rebase -i HEAD~4

# Before: pick A, pick B, pick C, pick D
# After: pick B, pick A, pick D, pick C (reordered)

Using exec During Rebase

Run tests at each commit to ensure nothing breaks.

$ git rebase -i HEAD~4

pick a1b2c3d Add validation
exec npm test
pick b2c3d4e Add form
exec npm test

Cherry-pick & Selective History

Basic Cherry-pick

Apply a specific commit from one branch to another.

$ git cherry-pick a1b2c3d

# Applies commit a1b2c3d to current branch

Cherry-pick Multiple Commits

$ git cherry-pick a1b2c3d b2c3d4e c3d4e5f

# Applies three commits in order

Cherry-pick Range

$ git cherry-pick a1b2c3d..c3d4e5f

# From a1b2c3d up to c3d4e5f (excludes a1b2c3d)

$ git cherry-pick a1b2c3d^..c3d4e5f
# Includes a1b2c3d (add ^)

Cherry-pick with Conflicts

$ git cherry-pick a1b2c3d
CONFLICT (content): Merge conflict in app.js

$ git add app.js
$ git cherry-pick --continue

# OR abort
$ git cherry-pick --abort

Cherry-pick Without Committing

$ git cherry-pick -n a1b2c3d

# -n = --no-commit: applies but doesn't commit

Reflog & Recovery: Undo Almost Anything

What is Reflog?

The reference log records every HEAD change. Deleted commits are recoverable for ~30 days.

View Reflog

$ git reflog

a1b2c3d HEAD@{0}: rebase finished: returning to main
b2c3d4e HEAD@{1}: rebase: Fix typo
c3d4e5f HEAD@{2}: rebase: Add validation
d4e5f6g HEAD@{3}: checkout: moving from feature to main

Recover Lost Commits

Scenario: Accidentally hard reset
$ git reset --hard HEAD~3
# Oops! Lost 3 commits

$ git reflog
a1b2c3d HEAD@{0}: reset: moving to HEAD~3
b2c3d4e HEAD@{1}: commit: Important fix
c3d4e5f HEAD@{2}: commit: Feature X
d4e5f6g HEAD@{3}: commit: Initial feature

$ git reset --hard HEAD@{3}

✓ Recovered all lost commits!

Recover Deleted Branch

$ git branch -D feature-branch
# Oops! Deleted by mistake

$ git reflog
a1b2c3d HEAD@{5}: checkout: moving from feature-branch to main

$ git checkout -b feature-branch a1b2c3d

✓ Branch recovered!

Track Specific Branch Reflog

$ git reflog feature

# Shows reflog for feature branch only

Rewriting History Safely

The Sacred Rule

NEVER rewrite history on shared branches. Only rewrite on branches you own, before pushing or with team coordination.

Changing Commit Messages

Last Commit Message

$ git commit --amend -m "Correct message"

# Changes the message of the last commit

Older Commit Message

$ git rebase -i HEAD~3

reword a1b2c3d Fix typo
pick b2c3d4e Next
pick c3d4e5f Final

# Editor opens for each reword

Amend Last Commit (Add Files)

$ git add forgotten-file.js
$ git commit --amend --no-edit

# Adds file to last commit, keeps message

Splitting a Commit

Split commit into two
$ git rebase -i HEAD~1

edit a1b2c3d Add feature + refactoring

$ git reset --soft HEAD~1
$ git add feature-files.js
$ git commit -m "Add feature"
$ git add refactoring-files.js
$ git commit -m "Refactor"
$ git rebase --continue

Debugging with Bisect

Finding Bad Commits

git bisect does binary search to find which commit introduced a bug.

Basic Bisect Workflow

Find which commit broke tests
$ git bisect start

$ git bisect bad
# Current commit is broken

$ git bisect good v1.0
# v1.0 was working

# Git jumps to middle commit
$ npm test
$ git bisect bad  # or good

# Repeat until converged

$ git bisect reset

✓ Found the culprit commit!

Automated Bisect

$ git bisect start
$ git bisect bad HEAD
$ git bisect good v1.0
$ git bisect run npm test

# Fully automated!
# Exit 0 = good, exit 1 = bad

Git Worktrees: Multiple Branches Simultaneously

Why Worktrees?

Worktrees let you check out multiple branches simultaneously in different directories. Better than constantly switching.

Create a Worktree

$ git worktree add ../feature-branch feature

# Creates ../feature-branch directory
# Checks out feature branch there
# Your current directory still has main

List Worktrees

$ git worktree list

/project/main           a1b2c3d [main]
/project/../feature     b2c3d4e [feature]
/project/../bugfix      c3d4e5f [bugfix]

Create Worktree from New Branch

$ git worktree add -b hotfix ../hotfix main

# Creates new branch 'hotfix' based on main
# Creates worktree for it

Delete Worktree

$ git worktree remove ../feature-branch

# Removes worktree and cleans up

Advanced Merging Strategies

Merge Strategies Overview

Strategy Description When To Use
recursive Standard 3-way merge (default) Most cases
resolve Simpler 3-way merge Simple merges
ours Always prefer current branch Discarding changes
theirs Always prefer other branch Accept all changes
octopus Merge multiple branches 3+ branches at once

Merge with Specific Strategy

$ git merge -s ours feature

# Merges but always keeps current branch version

Recursive Merge Options

$ git merge -X theirs feature

# On conflicts, prefer feature's version

Squash Merge

$ git merge --squash feature

# Applies all changes but doesn't create merge commit
# Changes are staged, ready to commit

No Fast-Forward Merge

$ git merge --no-ff feature

# Forces merge commit even if fast-forward possible
# Preserves branch structure

When to Use Terminal vs UI

Use VS Code UI When:

  • Simple merge/rebase operations
  • Committing and amending
  • Switching branches
  • Viewing history and diffs
  • Stashing work temporarily

Use Terminal When:

  • Interactive rebase (-i)
  • Advanced rebase (--onto, etc.)
  • Named stashes with messages
  • Bisecting to find bugs
  • Worktrees and complex workflows
  • Reflog recovery operations
Pro Tip: The integrated terminal in VS Code (Ctrl+`) gives you the best of both worlds — Git UI for visual operations and terminal for advanced commands, all in one window!

Solutions: Published Code + Local Changes

1. Configure VS Code for Safety

Create .vscode/settings.json in your project root to prevent Git chaos:

{
  // Disable auto-sync (manual control)
  "git.autofetch": false,
  "git.autorefresh": false,

  // Always show merge/rebase conflicts
  "git.ignoreMissingGitWarning": true,

  // Require confirmation for dangerous operations
  "git.confirmSync": true,

  // Use merge, not rebase by default
  "git.pullBeforeCheckout": true,

  // Show ignored files (be aware of what's tracked)
  "git.showIgnoredCountBadge": true
}

2. Configure Git Globally for Safety

Run these commands ONCE to set safe defaults:

# Always merge by default, never auto-rebase
$ git config --global pull.rebase false

# Require explicit auth (no sneaky auto-login)
$ git config --global core.askPass true

# Protect against accidental force push
$ git config --global push.default upstream

# Force confirmation before dangerous operations
$ git config --global core.protectNTFS true

# View all your settings
$ git config --global -l

3. Safe Workflow: Push → Modify → Commit → Push

The CORRECT sequence that works with VS Code
✓ Step 1: Start fresh
$ git pull origin main

✓ Step 2: Make changes locally
# Edit files in VS Code

✓ Step 3: Commit your changes
$ git commit -am "feat: new feature"

✓ Step 4: BEFORE push, sync with remote
$ git fetch origin
$ git status
# If says "ahead by 1", you're safe

✓ Step 5: Push safely
$ git push origin main

❌ NEVER:
- Don't amend after push
- Don't rebase after push
- Don't reset after push
- Don't force push

4. When VS Code Gets Confused: Recovery Commands

Problem: 'Your branch has diverged'

$ git fetch origin
$ git merge --no-ff origin/main
$ git push origin main

# This ALWAYS works - creates a merge commit

Problem: 'Commits lost' or 'changes disappeared'

$ git reflog
# Find your commit SHA in the list

$ git log --all --oneline --graph
# See ALL commits, including lost ones

$ git checkout -b recovery-branch a1b2c3d
# Go back to that commit and create branch

$ git merge recovery-branch main
# Merge recovered commits back to main

Problem: Merge/Rebase stuck in VS Code

$ git status
# If shows "rebasing" or "merging":

$ git merge --abort
# OR
$ git rebase --abort

# Returns to normal state
# Try again with: git merge --no-ff origin/main

Problem: Can't push - 'rejected by remote'

$ git pull --no-ff origin main
# This merges remote changes into your branch

$ git push origin main
# Now it should work

❌ DON'T do:
$ git push -f  (force push - dangerous!)
$ git rebase origin/main (rewrites history)

5. Prevention Checklist

Before You Push After You Push
✓ Run: git pull origin main ❌ Never amend commits
✓ Check: git status ❌ Never rebase commits
✓ Review: git diff origin/main ❌ Never hard reset
✓ Test: Make sure code works ✓ Only create NEW commits
✓ Commit: git commit -m "..." ✓ Pull first if error
✓ Push: git push origin main ✓ Use merge, not rebase
Key Rule: After push = READ-ONLY. Only add NEW commits. Never modify, rebase, or reset published commits. Remote is source of truth.
